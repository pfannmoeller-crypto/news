Kontext & Vorgeschichte. The map now reads Austin, Dallas, Houston, the Bay Area (with safety monitors), Miami, Orlando and Tampa — seven markets, up from four a month ago. Read as a map, the expansion looks aggressive: two new Florida cities added in three weeks. Read as a fleet, the picture inverts. Die Details. The two figures are not obviously reconcilable from public reporting, and that gap is itself the finding. The most likely explanation is a definitional one: BofA's 175 may count the total registered robotaxi-spec fleet, including vehicles still running with in-car monitors, undergoing validation, or staged for future markets, while Electrek's ~21 counts only vehicles Tesla is currently operating fully unsupervised on public roads. If that is right, both numbers can be true simultaneously and the real question is which one investors should be pricing — fleet capacity or fleet utilisation. What is not ambiguous is the Austin trend line: down from ~25 in late April to ~17 now, a genuine decline in the one market with the longest operating history, one year after the 22 June 2025 launch. Orlando and Tampa, by TechCrunch's account, have "fairly small operational areas" and no disclosed vehicle counts, which is consistent with a map-expansion strategy that is, as this week's coverage frames it, "cheap" relative to actually scaling a fleet.

Einordnung. The timing is the tell. A same-day announcement of two new markets, the day before an earnings call where robotaxi is the top shareholder concern, following a quarter in which the Say portal filled with demands to explain three consecutive guidance misses, reads as an answer engineered for the headline rather than the fleet report. That does not make it false — Orlando and Tampa are real, monitored deployments — but it does mean the city-count metric that Tesla and its bulls have used all year is now demonstrably decoupled from the vehicle-count metric that determines actual revenue. For anyone modelling robotaxi economics, city count should be weighted near zero until Tesla or a credible third party publishes a reconciled, monitored-vs-unsupervised fleet number by market.

SPACEXAISecurity15 – 16 Jul 2026

Last week's "trust repair" open-sourcing produced a working Grok Build exploit within 24 hours — and SpaceXAI told the researcher the client-side bug was "out of scope"

SpaceXAI open-sourced the Grok Build client on 15 July as its remedy for the data-exfiltration scandal this briefing covered last week. Within 24 hours, security researchers at SlowMist had published two working attack chains against the newly public code, one of which needs only three lines of JSON to bypass every permission gate in the tool. When the finding was submitted through HackerOne, SpaceXAI's stated response was that "client side issues on grok-build-cli are considered out of scope."

The first chain exploits a misclassification: Grok Build treats the command cargo check as read-only and auto-approves it, but that command executes arbitrary code via Rust's build.rs scripts — meaning a malicious project file can achieve code execution with zero user interaction. The second exploits a parser function that ignores a project's trust state entirely, so a three-line .claude/settings.json file setting "defaultMode": "bypassPermissions" silently disables every safety check. Because Grok Build shares its configuration format with Claude Code's CLI, a single malicious file can compromise both tools — though the researchers note Claude Code documents its own risk boundaries, while Grok Build's bypass was silent.

Kontext & Vorgeschichte. Last week's edition credited SpaceXAI's open-sourcing decision as "genuinely the correct response" to the Grok Build data-exfiltration scandal, while flagging that its unverifiable deletion promise deserved less credit. That judgement now needs revising in one direction: open-sourcing did make the code auditable, which is exactly why SlowMist could find and publish these flaws within a day — the mechanism worked as designed. What did not work is everything downstream of the audit. A security researcher doing exactly what open-sourcing is supposed to enable submitted a real, exploitable vulnerability through the company's own disclosed channel, HackerOne, and received a response that the class of bug — client-side issues in the CLI tool at the centre of the entire scandal — does not qualify for review.

Die Details. The technical substance is severe on its own terms, independent of the corporate response. The cargo check misclassification means Grok Build's permission model conflates "this command doesn't write files" with "this command is safe," when in fact Rust's build system routinely executes arbitrary code as a side effect of a nominally read-only check — a well-known category of risk in the Rust tooling ecosystem that a coding-agent security review should have caught before shipping, let alone before open-sourcing as a trust-restoration measure. The bypassPermissions flaw is worse in kind: SlowMist's analysis found five independent, disconnected trust modules inside the tool, meaning the vulnerability is not a single patchable line but an architectural absence of a unified trust model across code paths. Three lines of attacker-controlled JSON defeating the entire permission system is the kind of finding that would normally trigger an emergency patch and public disclosure timeline, not a scope classification.

Einordnung. For SPCX holders, this is a more damaging data point than the original incident, because it strips the credibility from the specific remedy the company offered. An enterprise buyer evaluating Grok Build post-scandal was told: trust us because the code is now public and auditable. The code being public and auditable produced a real 0-day in 24 hours, and the company's engineering response — as distinct from its public-relations response — was to decline responsibility for the exact category of bug the scandal was about. That is a worse signal for enterprise procurement than the original incident, which could at least be framed as an unintentional beta-stage failure; declining to fix an actively exploited class of vulnerability after promising reform is a choice. Set against Bloomberg's reporting last week that SpaceXAI's internal mandate is explicitly to "match Claude" on a compressed timeline, this reads as further evidence that the organisation is optimising for release velocity over the security maturity its enterprise pitch depends on.

Watch-Items. (1) Whether SpaceXAI reverses the HackerOne "out of scope" classification under further public pressure. (2) Any confirmed real-world exploitation of either attack chain, as opposed to responsible-disclosure proof-of-concept. (3) Whether Anthropic or other CLI-tool vendors sharing the .claude/settings.json format issue a joint advisory. (4) Any enterprise customer publicly pausing or cancelling a Grok Build deployment over this specific finding. (5) Whether the unified-trust-model fix SlowMist recommends appears in a subsequent release, and on what timeline.
